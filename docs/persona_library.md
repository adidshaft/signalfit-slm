# Persona Library Spec (v0.1)

Defines the synthetic personas and the stochastic daily-history model behind every
training/eval context. Goal: numbers that are **causally coherent** — when an answer
says "your HRV is down after two late nights," the seeded history actually shows it.

## 1. Persona grid

~120 training personas sampled (not full cross-product) from:

| Dimension | Values |
|---|---|
| age_band | 18-24, 25-29, 30-34, 40-44, 50-54, 60-64 |
| biological_sex | male, female, other/unspecified |
| primary_goal | general_fitness, endurance_event, strength_gain, fat_loss, improve_recovery, stress_management |
| fitness_level | beginner, intermediate, advanced (drives HRV/RHR priors and strain patterns) |
| provider_mask | wearable_full, ring_no_strain, platform_aggregate, watch_load_native, tracker_azm, manual_only |
| data_tier | rich (60d history, >90% coverage), typical (30-45d, 70-90%), sparse (7-21d, <70% — "learning" states) |
| behavior_profile | 2-4 recurring behaviors with TRUE effect sizes (see §3) |

Sampling constraints: every provider_mask ≥12 personas; every task category
reachable from ≥30 personas; ≥15% sparse-tier; cycle-tracking enabled for a subset
of female personas only (opt-in realism). Eval personas live in a disjoint
namespace (`eval-p***`) with different sampled parameters — no reuse.

## 2. Latent physiology model (per persona)

Draw once per persona, then generate 30–60 days:

- `hrv_mu` lognormal by age/fitness (e.g. ln-space mean so median ≈ 35–95 ms),
  `hrv_cv` 6–12%; daily HRV = lognormal AR(1) (ρ≈0.55) around hrv_mu.
- `rhr_mu` 45–70 bpm by fitness; daily RHR = normal AR(1), negatively correlated
  with same-day HRV (ρ≈-0.5).
- `sleep_need_h` 7.2–9.0; nightly sleep = need + weekday effect (weekend +25 min)
  + noise; bedtime jitter by persona conscientiousness.
- Weekly strain template by goal (e.g. endurance: 2 hard, 2 moderate, 1 long, 2
  easy/rest) with adherence noise; strain on the 0–21 scale for masks that have it.
- Recovery = the SignalFit reference formula: logistic blend of z-scores of HRV
  (weight ~0.5), inverted RHR (~0.25), sleep (~0.25) vs the persona's own trailing
  baseline — same family as the verified Atria model, so recovery_contributors
  directions are automatically consistent with the inputs.

## 3. Behavior covariates (ground-truth effects)

Behaviors are seeded events that REALLY move next-day numbers (effect ± noise):

| Behavior | True effect (next day unless noted) |
|---|---|
| alcohol (2+ drinks) | HRV −12%, RHR +4 bpm, recovery −10 pts, REM −20% |
| late_caffeine (after 15:00) | sleep −25 min, efficiency −4 pts |
| high_stress day | HRV −6%, sleep −15 min |
| late_meal | efficiency −3 pts |
| consistent_bedtime streak ≥5d | sleep +15 min, recovery +3 pts |
| illness episode (rare, 2-4 days) | HRV −25%, RHR +8 bpm, resp rate +2 bpm — used for possible_illness_pattern safety cases |
| overreach block (rare) | ACWR >1.5, monotony >2.0, HRV drifts down — overtraining cases |

Journal/nutrition logging is itself probabilistic (personas under-log), so
habit_insights only appear when the Atria-style gate would pass (≥5 tagged + ≥5
comparison days, effect ≥3 pts, Welch p<0.10) — matching real provider behavior.

## 4. Coherence invariants (checked by the critic, generated by construction)

1. baselines/trends are the actual stats of the seeded series (recompute, don't state).
2. `recovery_contributors` directions match the z-scores that produced recovery.
3. `anomalies` only when the seeded series really breaches (e.g. HRV < mean−1·sd
   for N days). 4. `missing_fields` ↔ mask nulls exactly. 5. `allowed_numbers`
   enumerates every context numeral. 6. Safety-case vitals actually exhibit the
   pattern the flag names (no "possible_illness_pattern" over healthy numbers).

## 5. Question sampling

Per example: sample task_category per the data-plan distribution, then a question
template family (12–20 per category) × persona voice (terse/chatty/typo-prone/
anxious/data-nerd) × optional emotional framing. Templates live with the generator
prompt; eval uses disjoint families.

## 6. Implementation note

The simulator itself is pipeline code (`scripts/`), deliberately NOT implemented
yet per the no-implementation-code constraint. This spec is the contract for it.
