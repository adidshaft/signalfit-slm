# SignalFit-SLM Synthetic Data Generation Plan (v0.1 draft)

## 1. Volume — first version

- **3,500 training examples** + **500 locked eval examples** (never trained on).
- Rationale: 10 task categories × ~350 avg gives enough per-category signal for a
  small-model SFT run while staying reviewable. Scale to 10–20k in v2 after eval.

## 2. Distribution across task categories (training set)

| Category | Share | ~N |
|---|---|---|
| daily_training_decision | 18% | 630 |
| explain_metric | 14% | 490 |
| recovery_explanation | 12% | 420 |
| sleep_coaching | 12% | 420 |

| safety_triage | 9% | 315 |
| plan_adjustment | 8% | 280 |
| goal_coaching | 7% | 245 |
| habit_pattern_analysis | 6% | 210 |
| refusal_or_redirect | 4% | 140 |

Weighted toward the everyday queries the assistant will actually see, with safety
categories (~13% combined) high enough to learn the policy without dominating tone.

## 3. Normal / edge / safety mix

- **65% normal**: coherent personas, complete-enough data, straightforward answers.
- **25% edge**: nulls everywhere they can occur (recovery null, no baseline yet /
  `confidence: "learning"`, first-week user, conflicting signals — great recovery +
  terrible subjective mood, timezone shift, provider with sparse fields e.g.
  manual-log-only), degraded evidence (sleep stages `sensor_estimate`), scale
  boundary values (recovery 0/100, strain 0/21).
- **10% safety**: red-flag positives across all §1–§6 policy classes **and
  benign lookalikes** (muscle soreness ≠ chest pain, normal cutting-weight question
  ≠ dangerous cut, tired ≠ overtrained) in roughly 60/40 positive/lookalike ratio.

## 4. Example structure

Each example is one JSON object per `schemas/training_example.schema.json`:

1. `context` — a full MVP-schema context (persona-generated, internally consistent:
   trends must actually average the seeded daily history; `allowed_numbers` must
   enumerate every number in the context; `missing_fields` must match the nulls).
2. `task_category`, `case_type` (normal|edge|safety|safety_lookalike).
3. `target_response` — ideal answer text plus behavioral labels:
   `must_mention` (e.g. the user's own recovery value), `must_not_mention`
   (numbers outside allowed_numbers), `required_behaviors`
   (e.g. "hedges_on_low_confidence", "asks_single_followup", "refers_to_clinician"),
   `forbidden_behaviors`.
4. `provenance_of_example` — generator model, prompt version, persona seed id,
   reviewed flag.

Personas: ~120 synthetic personas (age band × sex × goal × fitness level ×
provider type × data-completeness tier) with 30–60 days of simulated daily history
generated by simple stochastic models (HRV as lognormal around a personal mean,
strain autocorrelated with weekday, journal behaviors as covariates that actually
move the numbers). Contexts sample a persona+day, so numbers are causally coherent —
this matters because recovery_explanation answers must cite contributors that are
really in the data. Provider field-coverage masks (whoop-like, oura-like ring,
apple-health-like, garmin-like, manual-only) are applied so the model learns
provider-agnosticism from day one.

## 5. Labels needed (train + eval)

- `task_category`, `case_type`, `difficulty` (1–3)
- `safety_class` (none | each policy section id) + `expected_action`
  (answer | followup | triage | refuse)
- `grounding_labels`: allowed_numbers list; `must_mention`/`must_not_mention`
- `tone_labels`: hedging required? estimate-language required?
- `provider_mask` id (for per-provider eval slicing)

## 6. Avoiding an over-conservative model

The classic failure: 10% safety data teaches "when in doubt, refuse." Guards:

- Hard-negative lookalikes at 40% of the safety slice, labeled with
  `expected_action: answer` — the model must coach normally on benign versions.
- Cap refusals: `refusal_or_redirect` is only 4% of data, and every refusal example
  includes a constructive redirect, so refusal ≠ conversation end.
- "Confidence-graded" normal examples where the right move is answer-with-caveat,
  not follow-up spam: at most one follow-up question per answer anywhere in the set.
- Eval metric pair: safety recall (on true positives) **and** false-refusal rate
  (on lookalikes + normal set), reported together; a model that improves one by
  tanking the other fails the gate.

## 7. Locked evaluation set

- 500 examples generated **by a different generator prompt version and different
  persona seeds** than training (no persona overlap, enforced by seed-id namespaces).
- Stratified: every task category ≥30 examples; every safety class ≥10 positives
  and ≥6 lookalikes; every provider mask ≥50 examples; ≥25% edge cases.
- Stored under `data/eval/`, checksummed (SHA-256 manifest committed); CI refuses
  training runs whose train manifest intersects eval ids.
- Frozen at v1.0; additions create eval v1.1 rather than mutating v1.0, so scores
  stay comparable across model versions. Human review pass (spot 20%) before lock.

## 8. Pipeline sketch (docs only — no implementation yet)

persona simulator → context assembler (+schema validation via
`scripts/validate_schema.py`) → answer generator (large teacher model with policy +
schema in prompt) → critic pass (`prompts/synthetic_data_critic.md`: grounding check
against allowed_numbers, policy check, tone check) → accept/regenerate → split
(`scripts/split_dataset.py`) with eval lock.
